TypeScript and Module Federation sit on opposite sides of a build-time / runtime gap. The compiler needs types for `import('remote/Button')` *at build time* in the host. The actual module doesn't exist until *runtime*, when a remote you don't necessarily build hands it to you. Closing that gap is the job of type declarations — either hand-written or generated by the federation plugin.

## The build-time / runtime gap

When the host's `tsc` runs, the remote may not be built, may not be on disk, may belong to another team's repo. TypeScript needs *something* to look at for `'remote/Button'` to be a known module. Options:

1. Declare an ambient module yourself (manual, brittle).
2. Have the remote emit `.d.ts` files and the host consume them (`@mf-types`, automated).
3. Type as `unknown` and validate at runtime with a schema (last resort).

## Option 1: hand-written ambient declarations

In a `remote.d.ts` (or `src/types/remotes.d.ts`) anywhere TypeScript will pick up:

```ts
declare module 'remote/Button' {
  import type { ComponentType } from 'react';
  export const Button: ComponentType<{
    label: string;
    onClick?: () => void;
  }>;
}

declare module 'remote/mount' {
  export function mountButton(el: HTMLElement, props: { label: string }): () => void;
}
```

Pros: zero tooling, full control.
Cons: drift. The remote's source can change without the host noticing until runtime.

Use this when:
- You own both sides and the contract is small.
- You're prototyping.
- You want a *narrowed* type that hides remote internals.

## Option 2: `@mf-types` from the enhanced plugin

`@module-federation/enhanced` generates `.d.ts` files from each exposed module at build time and packages them into a tarball. The host fetches that tarball (over HTTP, from disk, or via build hook) and unpacks it into a local `@mf-types/` folder that's added to `tsconfig`.

```cjs
// remote rspack config
new ModuleFederationPlugin({
  name: 'remote',
  exposes: { './Button': './src/Button' },
  dts: {
    generateTypes: { compilerInstance: 'tsc' },
  },
});

// host rspack config
new ModuleFederationPlugin({
  name: 'host',
  remotes: { remote: 'remote@http://localhost:3001/remoteEntry.js' },
  dts: {
    consumeTypes: { remoteTypeUrls: { remote: 'http://localhost:3001/@mf-types.zip' } },
  },
});
```

Then in `tsconfig.json`:

```ts
{
  "compilerOptions": {
    "paths": {
      "remote/*": ["./@mf-types/remote/*"]
    }
  }
}
```

Now `import { Button } from 'remote/Button'` is fully typed and stays in sync with the remote's source.

Pros: automated, always-current, types are generated from the same source the remote ships.
Cons: extra build step, your host CI needs network access to the remote types endpoint (or a checked-in snapshot).

## Snapshotting types in CI

For deterministic builds without a network hop, commit the generated types to the host repo and refresh them via a scheduled job or a PR bot. Treat the snapshot as a contract — diffs in `@mf-types/` show up in code review the way `package-lock.json` diffs do.

## Typing dynamic remotes

If the URL is unknown at build time (e.g. plugin marketplace), you can't generate types ahead of time. Options:

- Declare a *base* contract every plugin implements:

  ```ts
  declare module 'plugin/*' {
    export function activate(api: PluginAPI): void;
    export function deactivate(): void;
  }
  ```

- Validate at runtime with Zod or similar before treating the module as that shape.

## Common patterns

| Pattern | Type approach |
| --- | --- |
| Single exposed component | `declare module 'remote/Button'` with the props type |
| Mount function | `declare module 'remote/mount'` with `(el, props) => () => void` |
| Hooks bundle | `declare module 'remote/hooks'` re-exporting each hook |
| Many exposes, two teams | `@mf-types` generated + consumed |
| Unknown plugins | Base contract + runtime validation |

## Gotchas

- `@mf-types` types describe the *current* remote build. A live production remote may be older. If you're rolling out a major, the types in the host may promise APIs the deployed remote doesn't have yet. Sequence: deploy remote, refresh types in host repo, then deploy host.
- `import type` is free — it's erased — but `import()` for code is not. Be precise about which one you're doing.
- Module resolution gotcha: TypeScript needs `moduleResolution: "bundler"` (or `node16`/`nodenext` with appropriate setup) for `paths` mapping to behave with federated modules.
- If you check in `@mf-types/` it should be in `.prettierignore` and `.eslintignore` to avoid noisy diffs.
- Hand-written `remote.d.ts` and generated `@mf-types/` shouldn't both declare the same module — TS will merge them and you'll get conflicts.
